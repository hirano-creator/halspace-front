# TimeCalc 勤怠管理システム リニューアル企画書

- 作成日: 2026-07-16
- 対象: TimeCalc（株式会社ヒラノ グループ勤怠管理システム）
- 本番: https://timecalc-hirano.netlify.app

---

## 1. 目的・背景

TimeCalcは当初「Square タイムカードのCSVを取り込み、勤務時間と金額を計算する少人数向けツール」として構築した。その後セルフ打刻（QRコード・GPS対応）まで実装済みである。

今回、計画変更により次の方針へ転換する。

> **グループ会社・複数部署をまたぐ約100名の勤怠を、このアプリのタイムカード機能で一元管理する。**

要望の全体像:

| 要望 | 本企画での扱い |
|---|---|
| このアプリで勤怠管理を実施 | 打刻を主経路とする運用に転換 |
| タイムカード機能（PC・スマホ） | 実装済み。UI/UXを全面改善（§5） |
| 100人程度の管理 | データモデル・一覧画面・無料枠の観点で対応（§4, §7） |
| 白ベースのシンプルなStripe風デザイン | デザイン刷新（§5）※「strip風」はStripe風と解釈 |
| グループ会社・部署 | 会社→部署の2階層に拡張（§4） |
| 金額は不要 | UI非表示（データは温存、§4） |
| 出勤・退勤・遅刻・早退・外出 | 打刻種別の拡張＋自動判定（§3） |
| 理由の記入 | 打刻時・後入力の両対応（§3） |
| 開発コスト以外ゼロ円 | Netlify＋Neon無料枠を継続（§7） |
| スマホ対応 | モバイルファーストで刷新（§5） |

## 2. 方針: 作り直しではなく「大改修」とする

**結論: 一から作り直さない。** 根拠は以下の通り。

| 観点 | 評価 |
|---|---|
| 既存カバー率 | 打刻（PC/スマホ/QR/GPS）・中抜け合算・社員/部署管理・権限・月次集計・CSV入出力・認証は実装済み。新規は「会社階層」「外出打刻」「遅刻早退判定」「理由入力」「マイページ」「デザイン」のみ |
| アーキテクチャ適合 | 本システムの核である「計算結果をDBに保存せず、表示時に設定から算出する」設計は、遅刻・早退の自動判定追加に最適。判定ロジックを純粋関数に足すだけで**過去データにも遡って適用**される |
| デプロイ資産 | Netlify＋Neon無料枠での本番構成（webpackビルド固定、リージョン最適化、DBウォームアップ）は試行錯誤の末に確立したもの。作り直すと全て再構築になる |
| スケール | 100人×1日4打刻 ≈ 400書込/日はPostgreSQLには誤差。作り直しの理由にならない |

作り直しが正当化されるのは既存設計が根本的に合わない場合だが、今回の変更は「追加＋非表示化＋デザイン刷新」であり、アーキテクチャはそのまま活きる。

## 3. 機能要件

### 3.1 打刻（タイムカード）

打刻ボタンを現行の2種から**4種**に拡張する。

| ボタン | 動作 | 勤怠上の扱い |
|---|---|---|
| 出勤 | その日最初の勤務開始 | 勤務時間の起点 |
| 退勤 | その日の勤務終了 | 勤務時間の終点 |
| 外出 | 勤務の中断 | **外出〜戻りの時間は勤務時間から除外**（休憩と同様に差し引く） |
| 戻り | 外出からの復帰 | 勤務再開 |

- 外出時間の除外は、既存の中抜け合算ロジック（退勤→再出勤の空白を休憩に合算する `deriveDailyFromEvents`）を流用する。
- QRコード打刻（店舗掲示・日替わりトークン）、GPS判定（部署座標＋許容半径、ユーザー単位ON/OFF）は現行のまま維持。

### 3.2 遅刻・早退（自動判定＋理由入力）

- 所定の始業・終業時刻（設定画面の勤務ルール）と打刻を比較し、**システムが自動で「遅刻」「早退」と判定**してバッジ表示する。手動フラグ立ては不要。
- 判定は表示時計算（保存しない）。所定時刻を変更すれば過去分の判定も追随する。
- **理由の記入**: 本人が打刻時にその場で入力できるほか、後から本人・管理者が日次画面で追記・修正できる。外出の理由（私用・通院など）も同じ仕組みで記入する。

### 3.3 マイページ（本人の勤怠確認）★追加要件

全社員が**ログイン後すぐに自分の勤怠を確認できる**ページを新設する。

- 本日の状態（出勤中／外出中／退勤済み）と本日のタイムライン
- 当月の日別一覧: 出勤・退勤時刻、勤務時間、遅刻・早退・外出のバッジ、理由
- 月の合計勤務時間・出勤日数・遅刻/早退回数
- 月切り替えで過去分も閲覧可能
- 押し忘れ（未退勤）のアラート表示と、対象日からの**修正申請**の導線（§3.6）
- スマホでの閲覧を最優先にレイアウト（管理者用の全社一覧とは別画面）

### 3.4 組織: グループ会社・部署

- **会社（Company）モデルを新設**し、「会社 → 部署（店舗） → 社員」の2階層にする。
- 管理者は全社横断で閲覧・管理。集計・一覧・CSV出力は会社／部署で絞り込み可能。
- 既存データは移行時にデフォルト会社（株式会社ヒラノ）へ紐付け、無停止で移行する。

### 3.5 金額機能・CSV取込の扱い

- **金額（時給・残業代・支給額）はUIから非表示にする**。データ（hourlyWage・計算ロジック）は削除せず温存し、設定でON/OFFできる形にする（将来復活可能）。
- **Square CSV取込は残す**（過去データの投入・併用期間のため）。取込画面は管理者メニューの奥に配置する。

### 3.6 打刻の押し忘れ・修正対応 ★追加要件

- **押し忘れの検知**: 退勤打刻がないまま日をまたいだ勤怠は「未退勤」として、本人のマイページと管理者の勤怠一覧の両方でアラート表示する。
- **修正申請（デフォルトの修正経路）**: 本人がマイページから対象日を選び、「正しい時刻＋理由」を添えて修正申請を出す。管理者・マネージャーが承認すると勤怠に反映される。却下時は理由を返せる。
- **直接修正**: 管理者・マネージャーは申請を介さず直接修正できる（現行の手動修正を継承）。さらに§3.7のスタッフ単位設定で、信頼できるスタッフ（店長など）に「本人直接修正可」を個別付与できる。
- **修正履歴**: すべての修正・承認について「誰が・いつ・何をどう変えたか」を記録し、対象日の詳細で確認できるようにする（勤怠の信頼性担保）。

### 3.7 スタッフ単位の機能設定 ★追加要件

ロールで基本権限を決めつつ、**アカウント（スタッフ）単位で機能をON/OFF・切り替えできる**ハイブリッド方式にする。既存の「GPS判定のユーザー単位ON/OFF（gpsCheckEnabled）」と同じ考え方を一般化する。

| 設定項目（例） | 選択肢 | 既定値 |
|---|---|---|
| GPS打刻判定 | ON / OFF | ON（既存） |
| 打刻修正 | 申請のみ / 本人直接修正可 / 不可 | 申請のみ |
| QRコード経由のみ打刻許可 | ON / OFF | OFF |
| マイページの月次集計表示 | ON / OFF | ON |

- 設定は社員編集画面に「機能設定」セクションとして追加。将来項目を増やしても画面・データ構造を変えずに済む形（JSON上書き方式）にする。
- 権限判定は現行どおり `can()` に集約し、「ロールの基本権限 → スタッフ単位の上書き」の順で解決する。

### 3.8 権限

現行の4ロール（管理者／マネージャー＝自部署のみ／社員／パート）と表示名変更機能を維持。マイページは全ロール、全社一覧・設定は権限に応じて表示。個別の機能差は§3.7のスタッフ単位設定で吸収する。

## 4. データモデル変更案

```
Company（新設）
  └─ Department（既存に companyId 追加）
       └─ User（既存のまま）

ClockEvent.type: "IN" | "OUT" に "OUT_START"（外出）| "OUT_END"（戻り）を追加
ClockEvent.reason: 打刻時の理由（新設・任意）
Attendance.lateReason / earlyLeaveReason（日単位の理由。既存noteとは分離）

CorrectionRequest（新設）: 修正申請（対象日・修正内容・理由・状態・申請者・承認者）
AttendanceLog（新設）: 修正履歴（誰が・いつ・何をどう変えたか）
User.featureOverrides（新設・JSON）: スタッフ単位の機能設定の上書き値
```

- 遅刻・早退の判定結果、外出の合計時間などの**計算値は一切保存しない**（現行設計を堅持）。
- マイグレーションは追加のみで後方互換。既存の打刻・勤怠データはそのまま使える。

## 5. 画面・デザイン

### 5.1 デザイン方針: 白ベース・Stripe風

- 白背景＋薄いグレーの細ボーダー、広めの余白、インディゴ〜バイオレット系のアクセント1色
- 影は最小限、角丸は控えめ、タイポグラフィで階層を作る
- 全画面モバイルファースト（打刻とマイページはスマホ利用が主、管理画面はPC主体だがスマホでも崩れない）

### 5.2 打刻画面の刷新 ★追加要件（現状のUI/UXが悪い）

現状は「名前＋状態テキスト＋ボタン2つ」の簡素なカードのみ。以下に刷新する。

- **現在時刻のリアルタイム表示** ★追加要件: 画面中央に大きなデジタル時計（時:分:秒、秒単位で更新）＋日付・曜日。「時計を見ながら打刻する」タイムレコーダーの体験を再現する
- 状態をカラーで明示: 未出勤（グレー）／出勤中（グリーン）／外出中（アンバー）／退勤済み（ブルー）
- 4つの大型ボタン（出勤・退勤・外出・戻り）。状態に応じて押せるボタンだけを活性化し、誤打刻を防ぐ
- 打刻成功時は打刻時刻を大きく表示するフィードバック（例:「出勤 09:02 を記録しました」）＋理由入力への導線（遅刻判定時は自動でプロンプト）
- 本日のタイムラインを時刻・種別バッジ付きで表示
- スマホで親指が届く下半分にボタンを配置、タップ領域は最低44px

### 5.3 画面構成（改修後）

| 画面 | 対象 | 内容 |
|---|---|---|
| 打刻 | 全員 | §5.2の刷新版 |
| マイページ | 全員 | 自分の勤怠履歴・月次集計（§3.3、ログイン後のデフォルト画面） |
| 勤怠一覧 | 管理者・マネージャー | 会社/部署/氏名/月で絞り込む全社一覧。遅刻・早退・未退勤バッジ、理由列を追加。金額列は非表示化 |
| 修正申請 | 全員（申請）／管理者・マネージャー（承認） | 押し忘れ・誤打刻の修正申請と承認・却下。未処理件数をバッジ表示 |
| 社員管理 | 管理者 | 会社・部署の紐付け、スタッフ単位の機能設定（§3.7）。100人運用に向け検索・一括登録を強化 |
| 設定 | 管理者 | 勤務ルール／会社・部署管理／QRコード／ロール表示名／金額表示ON/OFF |

## 6. 段階計画

| フェーズ | 内容 |
|---|---|
| **Phase A: 機能** | Companyモデル追加・外出/戻り打刻・遅刻早退の自動判定・理由入力・マイページ新設・修正申請と承認・修正履歴・スタッフ単位の機能設定・金額非表示化 |
| **Phase B: デザイン** | Stripe風デザイントークン整備 → 打刻画面刷新（リアルタイム時計含む）→ 全画面へ展開・モバイル最適化 |
| **Phase C: 100人運用準備** | 社員一括登録（CSV）、一覧の検索・ページング、実データでの負荷・無料枠消費の確認、運用手順書 |

各フェーズ完了ごとにPlaywright E2Eで検証（`.claude/skills/verify` の手順）し、Netlifyへデプロイする。

## 7. 非機能・コスト（ランニングコストゼロの維持）

| 項目 | 現状/対応 |
|---|---|
| ホスティング | Netlify 無料枠。Functions呼出は100人運用でも試算 〜52k/月（枠125k）で余裕。使用量は月次で確認する |
| DB | Neon 無料枠（月100 CU-hrs）。現行のウォームアップは打刻ピーク（始業前 7:30 前後・退勤 18:00 前後）に合わせて時間帯を再配分する。枠内に収めるため深夜帯は数秒のコールドスタートを許容 |
| 100人規模の負荷 | 打刻 ≈ 400書込/日、閲覧を含めてもPostgreSQL・Next.jsには軽負荷。ボトルネックは無料枠の関数呼出数のみ（上記の通り余裕） |
| 追加費用 | **ゼロ**。有料サービス・外部APIは導入しない |

## 8. ソース管理・バックアップ

- デプロイは現行どおり **Netlify CLI直接デプロイ**を継続（GitHubのリモートがPublicリポジトリのため、ソース公開を避ける目的。金額ロジックを温存する以上、この方針は維持）。
- ただしPC故障によるソース喪失対策として、**TimeCalc専用のPrivateリポジトリ（無料）をバックアップ専用**に新設し、節目ごとにpushする。デプロイとは連動させない。
