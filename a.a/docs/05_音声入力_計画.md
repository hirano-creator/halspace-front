# a.a 音声入力 計画書

> 投稿のテキストを**音声で入力**できるようにする。スマホ第一のa.aで、現場の経営者が文字を打つ手間を減らすのが目的。
> 手法＝**Web Speech API（ブラウザ内蔵・無料・サーバー不要）**、対象＝**新規投稿・投稿編集・コメント**の3箇所、配置＝**「投稿する」ボタンの直下**（コメント欄のみバー内アイコン）。

---

## 1. 手法の選定

| 案 | 仕組み | 精度 | コスト | 実装量 | 採否 |
|---|---|---|---|---|---|
| **A. Web Speech API** ✅ | ブラウザ内蔵の`SpeechRecognition`でリアルタイム認識 | OS依存（Android/iOSのOS標準認識） | 無料 | フロントのみ・最小 | **採用** |
| B. サーバー側認識（Whisper/Gemini） | `MediaRecorder`で録音→APIへ送って文字起こし | 高精度・全ブラウザ同品質 | API従量課金 | solid-api改修が必要 | 将来拡張（§7） |

- 採用理由：meetlog（`meetlog/assets/js/pages/ml-edit.js:148-209`）で**実戦済みの同型実装**があり移植が最短。バックエンド変更ゼロで、精度不満が出たら案BへUIを変えずに差し替えられる。

### 対応ブラウザ

| 環境 | 対応 | 備考 |
|---|---|---|
| Android Chrome | ○ | 最も安定。主要ターゲット |
| iOS Safari (14.5+) | ○ | `webkitSpeechRecognition`。認識が数秒で切れやすい→`onend`自動再startで吸収 |
| PC Chrome / Edge | ○ | |
| Firefox | × | ボタン自体を非表示にする（機能劣化なし） |

- **HTTPS必須**（本番Cloudflare Pages・localhostはOK。LAN内IPでのスマホ確認は不可→本番かトンネル経由で確認）。
- マイク許可は初回タップ時にブラウザが確認ダイアログを出す（アプリ側の実装不要）。

## 2. UI配置

デザイン規約どおり白黒基調・コバルト（#1f48ff）は状態表示のみ。赤は使わない。アイコンは既存と同じ**インラインSVGラインアイコン**（stroke-width 1.6〜1.9）でマイク型を新規作成。

### 2-1. 新規投稿 `app/compose.html`（#body）

```
┌─────────────────────────────┐
│ 共有したいこと・気づきを書く…       │  ← textarea #body
└─────────────────────────────┘
(📷写真)(🎬動画)(📄資料)(◼マスク)(▦モザイク)   ← 既存chip行（変更なし）

[        投稿する        ]   ← .btn.primary（コバルト）既存
[   🎤 音声で入力        ]   ← .btn（白黒）新設 #voiceBtn
```

- 待機中：白黒 `.btn`＋マイクSVG＋「音声で入力」
- 認識中：ラベル「認識中… タップで停止」＋**コバルトのドットがパルス**（`.listening`）

### 2-2. 投稿編集 `app/post.html` renderEdit()（#ebody）

保存ボタン `#esave` の直下に compose と同一マークアップのボタンを配置。

### 2-3. コメント `app/post.html`（#cbody）

画面下部固定バー `.cbar` のため「ボタンを下に置く」レイアウトが取れない唯一の箇所。**送信ボタン`#csend`の左にマイクアイコンボタン**（`.cbar-send`と同寸）を置く：

```
┌──────────────────────────────┐
│ [コメントを書く…        ] 🎤  ➤ │  ← .cbar（画面下固定）

```

認識中はマイクアイコンがコバルト＋パルス。interim表示はバー上に小さくオーバーレイ。

## 4. 共通モジュール `AAVoice` の設計

meetlogの実装（`initVoiceRecognition`/`toggleVoice`）を汎用化。ページ側にロジックを書かず、3箇所すべて `attach()` 1呼び出しで済ませる：

```js
AAVoice.supported()   // SpeechRecognition対応判定
AAVoice.attach({ button, target, interimEl, onState })
// button   : トグルボタン要素（非対応ブラウザなら hidden にする）
// target   : input または textarea。final確定分をカーソル位置へ挿入
// interimEl: 認識途中テキストの表示先（targetは汚さない）
// onState  : (listening) => ボタンの見た目切替コールバック
```

挙動（meetlogから踏襲・調整）：

- `lang='ja-JP'`、`continuous=true`、`interimResults=true`
- `onresult`：`isFinal` の確定分だけを **targetのカーソル位置に挿入**（meetlogの末尾追記＋改行から変更。SNS投稿では不要な改行を入れない）
- `onend`：listening中なら自動 `start()`（iOS Safariの短時間切断対策）
- `onerror`：停止して interimEl にエラー表示（`not-allowed`＝マイク許可拒否は案内文言）
- 同一ページ内で複数attachした場合、録音は**同時に1つだけ**（他が開始されたら停止）

## 5. 状態遷移

```
[待機] --タップ--> [認識中] --タップ--> [待機]
                    │ onresult(final)  → targetへ挿入
                    │ onresult(interim)→ interimElへ表示
                    │ onend(自動)      → 再start（認識中を維持）
                    └ onerror          → [待機]＋エラー表示
```

ページ遷移・投稿送信時は必ず `stop()`（マイクを掴んだままにしない）。

## 6. テスト計画

1. **ローカル（PC Chrome）**：静的サーバー起動→`demo@a.a`ログイン→composeでマイク許可→発話で確定テキストが#bodyのカーソル位置に入る／interim表示／停止トグル／投稿送信で自動stop
2. **非対応ブラウザ（Firefox）**：ボタンが非表示になり他機能に影響しないこと
3. **スマホ実機（HTTPS＝本番 or トンネル）**：Android Chrome / iOS Safariで3箇所（投稿・編集・コメント）を確認。iOSで認識が数秒で切れても自動再開すること、ホーム画面追加（PWA）状態でも動くこと
4. **マイク許可拒否**：エラー文言が出て待機に戻ること

## 7. 将来拡張（精度不足が出た場合）

`MediaRecorder`で録音→solid-apiに `/api/aa/transcribe`（Whisper/Gemini文字起こし）を新設して差し替える。`AAVoice.attach()` のインターフェースは変えず、内部実装のみ切り替える設計にしておく（UIは共通）。
