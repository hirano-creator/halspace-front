# What'sNo スキル機能 ロードマップ

> このプロジェクト（スキル機能）を「これからどう育てるか」をまとめた計画書。
> 「作るべき完成形」の定義は [skill-spec.md](./skill-spec.md) を参照。本書は方向性・優先順位・運用を扱う。
> 最終更新: 2026-06-14

---

## 0. ビジョン

What'sNo を「ファイルを置く場所」から「**自然言語で指示すれば、現場の定型業務を片付けてくれる相棒**」へ。
ユーザーは1つの入力欄に普段の言葉を打つだけで、メール送信・共有・承認・タグ付けといった作業が、
**下書き→確認→実行**の安全な流れで完了する。さらに**自分専用のスキルを会話で作れる**。

育成の軸（メモリの育成パス）:
**レベル1（固定スキル）→ レベル2（マイスキル）→ レベル3（自然言語定義＋ツール自動選択・連鎖）**

## 1. 現在地（達成済み）

| 領域 | 状態 |
|------|------|
| ディスパッチャ型ルーティング | ✅ 1入力欄→複数スキル自動振り分け（`routeSkill`、JSON抽出方式） |
| 組み込みスキル | ✅ メール送信 / 共有リンク / 承認申請 / AIタグ の4種 |
| スキル基盤 | ✅ `SkillHandler` IF＋`SkillRegistry`（増やしやすい構造） |
| マイスキル | ✅ `wn_user_skills` CRUD＋フォームUI＋`UserDefinedSkillHandler` |
| 会話型ビルダー | ✅ 実現可否判定→提案→フォーム自動入力（`buildSkillFromChat`） |
| 実行ログ | ✅ `wn_skill_runs`（drafted/executed/canceled/failed/unmatched） |
| 安全モデル | ✅ 副作用は既存APIのみ・任意コード実行なし・全スキル確認必須 |

→ **おおむねレベル1＋レベル2に到達**。次はこれを「実運用で磨く」段階。

## 2. 短期（運用開始と磨き込み）

実際に現場で使ってもらい、ログを見て精度を上げるフェーズ。

### 2.1 運用に乗せる
- [ ] 本番デプロイ（API→フロント）後、4スキル＋マイスキルの実機確認。
- [ ] **ログを観察する習慣**: `wn_skill_runs` の `unmatched` 率・`canceled` 率・`confidence` 分布を見て、
      振り分け精度と「効いていないスキル」を把握する。

### 2.2 振り分け精度の改善
- [ ] `unmatched` が多い言い回しを `triggerHints` に追加してルーティング精度を上げる。
- [ ] 低 `confidence` 時の挙動を改善（「〇〇スキルでよいですか？」と確認してから実行）。
- [ ] 取り違え（誤スキル発動）が出たら、プロンプトのルールを調整。

### 2.3 新しい組み込みスキルの追加（低コストで価値が出る順）
既存の `GeminiService` メソッドを流用できるものから:
- [ ] **Q&A / Brain連携**（`answerWithChunkContext()` 流用、「この資料の〇〇を教えて」）
- [ ] **議事録整形・アクション抽出**（`formatMinutes()` / `extractActions()` 流用、MeetLog連携）
- [ ] **セマンティック検索**（`embed()`＋既存 `semanticSearch`）

### 2.4 UXの仕上げ
- [ ] `confirm()` で代用している実行前確認を、**統一した確認モーダル**に置き換え（draft内容を綺麗に表示）。
- [ ] メールスキルの `executed` 記録タイミングの見直し（現状は下書き提示時に記録する簡易実装）。
- [ ] スキル実行中のローディング表示・連打防止の強化。

## 3. 中期（賢さと運用性）

### 3.1 スキルの表現力
- [ ] **複数ファイル対象**（現状は選択先頭1件）。「選択した3件を共有して」等。
- [ ] マイスキルの **`conditions` 活用**（対象タグ・拡張子で発動可否を制御）。
- [ ] テンプレのプレースホルダ拡充（`{{file_name}}` に加え `{{today}}` `{{user_name}}` など）。

### 3.2 会話型ビルダーの強化
- [ ] 会話履歴を **セッション化**（現状はフロント保持・毎回送信）。Brainのセッション基盤と統合検討。
- [ ] 提案時に「**何が起きるか**」のプレビュー（このスキルを実行すると…）を会話内に表示。
- [ ] マイスキルの**共有/テンプレ配布**（会社内で「おすすめスキル」を配る）。

### 3.3 実行履歴UI・分析
- [ ] スキルバーから **履歴モーダル**（`wnGetSkillRuns`）。いつ・誰・どのスキル・status。
- [ ] 簡易ダッシュボード（よく使うスキル、成功率、未マッチTOP）。

### 3.4 運用・計測（KPI）
| 指標 | 見方 |
|------|------|
| スキル発動数 / 日 | 利用が根付いているか |
| `executed` 率 | 提案が実際に使われているか |
| `unmatched` 率 | 振り分け漏れ＝改善ネタ |
| マイスキル数 | ユーザーが自分で作っているか（レベル2の浸透） |
| 平均 `confidence` | ルーティングの確信度 |

## 4. 長期（レベル3：自律と連鎖）

### 4.1 ツール自動選択・連鎖
- [ ] `routeSkill` を **function calling 対応**に移行（JSON抽出から、堅牢なツール選択へ）。
- [ ] **複数ツールの連鎖**: 「共有リンクを作って山田さんにメールして」＝share→email を1指示で。
- [ ] プランナーが**手順（プラン）を提示**し、ユーザーが承認して順に実行。

### 4.2 自動実行レベル
- [ ] スキルごとに **確認の強さ**を設定可能に（低リスクは即実行、メール送信は必ず確認）。
- [ ] 「いつもこれでいい」を覚える（メーラー記憶 `wn_mailer_pref` の発想を各スキルへ）。

### 4.3 視野に入れるが当面非対象
- イベント駆動の自動起動（アップロード時・承認時にスキル自動実行）。
  → キュー基盤・権限設計が必要。**当面は手動起動のみ**を維持。

## 5. 横断テーマ（常に意識する）

### セキュリティ
- `base_tool` は組み込みキーの **allowlist 検証**を死守。テンプレは固定プレースホルダのみ展開（任意式評価なし）。
- 全データ `company_id` スコープ。承認等の権限チェックは既存APIに委ねる。
- 連鎖・自動実行を入れるほど「誤操作・誤送信」リスクが上がる → 確認UIと監査ログを厚く。

### コスト（Gemini）
- ルーティングは `generate()`（メインキー）に集約。Brain枠（Knowl回答用）は食い潰さない。
- send_email は本文をルーティング応答に含め**1回呼び出しに統合**済み。連鎖時も呼び出し回数を意識。
- `maxOutputTokens` の上限（現状512）に注意。長文を返すスキルは個別調整。

### LLM精度・評価ループ
- `wn_skill_runs` の `instruction`／`args_json`／`status` は**改善の宝庫**。
- 取り違え事例を集め、プロンプト／`triggerHints` を回す**改善サイクル**を回す。
- 将来は代表的な指示文の**回帰テスト（eval）**を用意し、プロンプト変更時のデグレを検知。

## 6. 技術的負債・宿題

| 項目 | 内容 | 優先 |
|------|------|------|
| 確認UI | `confirm()` 代用を統一モーダルへ | 中 |
| メール executed 記録 | 下書き提示＝executed の簡易実装を見直し | 低 |
| ルーティング堅牢性 | JSON抽出 → function calling（レベル3で） | 中 |
| 会話履歴 | フロント保持 → セッション化 | 中 |
| トークン上限 | 512 上限の見直し（長文スキル） | 低 |
| git運用 | `WhatsNo` 大文字小文字のindex統一（別途整理） | — |

## 8. 進め方の原則

1. **小さく出して、ログで磨く。** 大きな機能より、使われる1スキルを確実に。
2. **既存資産を最優先で流用。** 新規ロジックを書く前に `GeminiService`／既存APIを探す。
3. **安全第一。** 確認なしの自動実行は、誤操作リスクの低いものに限る。
4. **増やしやすさを保つ。** 新スキルは `SkillHandler` を1つ足すだけで済む構造を崩さない。
